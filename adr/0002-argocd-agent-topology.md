# ADR 0002: Argo CD Agent topology — managed mode, single Principal on hub-dc, Vault-backed PKI

## Status

Accepted — 2026-05-07

## Context

The four-cluster fleet (`hub-dc`, `hub-dr`, `spoke-dc`, `spoke-dr`) needs a
canonical mechanism for deploying application workloads from `lab-workloads.git`
onto the active workload cluster (`spoke-dc`) under the existing GitOps
contract. Three constraints, all stated by the owner during the 2026-05-07
working session:

1. **"Fully GitOps-driven; no manual `oc apply -f application.yaml`."**
   Implies workload `Application` resources must be generated by an
   `ApplicationSet`, never authored per-app. (See the related project ADR
   <https://github.com/zeshaq/brac-poc-openliberty-v5/blob/main/docs/adr/0007-no-per-app-application-yaml.md>.)
2. **"Spoke clusters use the Argo-based agent pull model."** Rules out the
   ACM `manifestwork` pull pattern (`cluster-config-managed-pull` style) for
   *application* workloads. ACM stays for cluster-config. The "Argo-based agent
   pull model" maps to the **Argo CD Agent** project, which Red Hat shipped
   GA in OpenShift GitOps 1.19 (Jan 2026) — verified installed at v1.20.3 on
   hub-dc.
3. **"lab-gitops is platform-only; apps belong in lab-workloads."** Confirmed
   in `gitops.html` and `operating-rules.html`. ApplicationSet definitions are
   platform tooling and live in `lab-gitops/argocd/applicationsets/`; app
   manifests live in `lab-workloads/components/apps/<base>/`. Industry
   consensus matches (Red Hat Developer "GitOps directory structure" article,
   Codefresh "How to structure your Argo CD repositories using ApplicationSets").

ADR-0001 (spoke-dr platform standby) is also load-bearing for this design:
since spoke-dr does not run application workloads in normal operation, the
Agent topology only needs to cover spoke-dc in the steady state. spoke-dr's
existing ACM-pull `cluster-config` reconciliation continues unchanged; an
Agent on spoke-dr is added at activation time during the DR drill, not now.

Three sub-decisions remained open after research narrowed the architectural
space, and were resolved on 2026-05-07 in the working session:

- **Operating mode** — Argo CD Agent supports Managed mode (hub owns the
  Application; agent on spoke executes) and Autonomous mode (each spoke owns
  its Applications independently; hub aggregates view-only).
- **Principal HA** — single Principal on hub-dc only, vs active Principal on
  hub-dc + warm-standby Principal on hub-dr sharing PKI material.
- **PKI / mTLS strategy** — `argocd-agentctl`-managed CA stored in Vault and
  projected via ESO; vs cert-manager with ClusterIssuer + parent CA per hub;
  vs self-signed per cluster.

## Decision

### Operating mode: **Managed**

The hub Argo CD on `hub-dc` owns the workload `Application`/`ApplicationSet`
resources. The Agent on `spoke-dc` runs the application-controller against
those Applications, reporting status back to the hub via the gRPC mTLS
control channel. The hub Argo CD UI shows live spoke status via the
resource-proxy.

This matches the canonical Red Hat 1.19 multi-cluster pattern, the
"single source of truth on the platform repo" principle that the rest of the
fleet already follows (`cluster-config`, `cluster-config-managed-pull`), and
the lab's "fully GitOps-driven" rule.

### Principal HA: **Single Principal on hub-dc**

One Principal CR on `hub-dc` in normal operation. `hub-dr` does **not** run a
Principal day-to-day. On a controlled `hub-dc` → `hub-dr` activation drill,
ACM Backup/Restore (Velero + the `cluster-backup` chart) restores the Argo CD
CRs, ApplicationSets, repo credentials, and the Principal CR onto `hub-dr`
from the latest hub backup. The agent on `spoke-dc` is configured with the
hub-side Principal endpoint (Route on hub-dc); on activation, DNS or the
manifest is updated to point at hub-dr's Route.

Rationale: ADR-0001's spoke-dr platform-standby decision committed the fleet
to **runbook-driven DR**, not automatic failover. Running a warm-standby
Principal on hub-dr — automatic Agent reconnect, dual PKI sync, dual
ApplicationSet replication — would introduce automatic failover on the hub
side while keeping it manual on the spoke side. Coherence wins; both hub and
spoke DR are deliberate, owner-approved drills.

### PKI / mTLS: **`argocd-agentctl`-managed CA in Vault, projected via ESO**

The Argo CD Agent CA is generated once with `argocd-agentctl init` and stored
in the lab Vault under
`secret/openshift-gitops/argocd-agent-pki/{ca-cert,ca-key,principal-server-cert,principal-server-key}`.
On hub-dc, an `ExternalSecret` (mounted into the `openshift-gitops` namespace
via the existing `rke2-vault` `SecretStore`) projects the Principal's
server cert + key into a Kubernetes `Secret` consumed by the Principal CR.
Per-agent client certs are issued by `argocd-agentctl` against the same CA
and seeded onto each spoke as one-time `Secret` objects via the Agent CR's
mTLS config.

Rationale: ADR-0004 (Vault-first secret policy) explicitly applies. The
existing ESO + Vault path is verified — the smoke `ExternalSecret` is Ready
on every cluster, including spoke-dc and spoke-dr. CA persistence in Vault
also makes the hub-dr DR path clean: the restored Principal on hub-dr reads
the same CA cert + key from Vault and immediately accepts the existing
agent client certs without re-enrollment.

## Consequences

**Accepted**

- Hub-side RTO during a `hub-dc` outage is the time to drive the ACM
  Backup/Restore drill on `hub-dr` — minutes-to-hour, not seconds. That's
  consistent with ADR-0001's deliberate-DR posture.
- The `lab-gitops` repo gets two new platform components:
  `components/platform/argocd-agent-principal` (activated on `hub-dc`) and
  `components/platform/argocd-agent-spoke` (activated on `spoke-dc`). spoke-dr
  doesn't get the spoke component until DR activation.
- A new ApplicationSet `lab-gitops/argocd/applicationsets/workloads-agent-managed.yaml`
  generates `<cluster>-workloads` Applications targeting the Agent on spoke-dc.
  The existing push-mode `workload-config.yaml` is retired (or coexists during
  transition) — see follow-up.
- A one-time owner step is required at install: run `argocd-agentctl init` to
  produce the CA material, write it into Vault, then deploy the Principal CR.
  Per-agent enrollment (`argocd-agentctl agent create <agent-id>`) issues the
  spoke client cert and writes it into a `Secret` ESO-projected onto the spoke.
- Spoke-dr's existing ACM-pull `cluster-config` reconciliation is unchanged.
  An Agent is added to spoke-dr at DR drill activation, not in normal ops.

**Avoided**

- Doubled Principal operational surface (PKI sync between hubs, dual
  ApplicationSet replication, two control-plane endpoints to monitor) for a
  POC.
- Inconsistent DR semantics where the hub Principal would auto-fail-over
  while the spoke workload requires runbook activation.
- A second trust chain (cert-manager-managed CA) running alongside Vault for
  what is conceptually one secret-management surface.
- The "self-signed per cluster" trap where DR rebuild on hub-dr requires
  re-enrolling every agent.

**Required follow-up** (out of scope for this ADR; tracked in the platform PR)

- Verify operator version on `spoke-dc` is ≥1.19 (hub-dc confirmed at v1.20.3
  on 2026-05-07; spoke-dc's RHACM-managed `acm-openshift-gitops` instance
  needs a version probe before Agent deploy).
- Author the platform PR on `lab-gitops`:
  `components/platform/argocd-agent-principal/`,
  `components/platform/argocd-agent-spoke/`,
  `argocd/applicationsets/workloads-agent-managed.yaml`, and the Vault KV
  bootstrap script.
- Decommission the existing push-mode `workload-config.yaml` ApplicationSet
  once the Agent path is verified.
- Document the install + per-agent enrollment owner steps as a runbook,
  most likely a new page in this wiki.

## Alternatives Considered

### Operating mode

- **Autonomous mode.** Each spoke runs an Argo CD that owns its Applications
  independently; the hub aggregates view-only. Rejected because:
  - It splits source-of-truth across clusters, contradicting the
    "lab-gitops is the live desired-state source" model documented in
    `gitops.html`.
  - It would require the workload `ApplicationSet` (or its generated
    Applications) to live in *every spoke's* local Argo, multiplying the
    governance surface.
  - The fleet's current pattern (`cluster-config-managed-pull`) is already a
    pull mechanism delivering hub-defined state to spokes — Managed mode is
    the same shape for workloads.

### Principal HA

- **Active + warm-standby Principal on both hubs.** Rejected because:
  - Doubles operational surface (PKI material projected to both hubs;
    ApplicationSets and Applications replicated; agent endpoints
    DNS-failover-aware) for marginal benefit at POC scale.
  - Inconsistent with ADR-0001's deliberate-DR posture for spoke-dr.
  - hub-dr's existing role per `topology.html` and `hub-dr.html` is
    "passive restore hub" — adding a live Principal contradicts that.
  - ACM Backup/Restore already covers the data path; adding a second
    Principal duplicates work the backup/restore controller is doing.

### PKI / mTLS

- **cert-manager with ClusterIssuer + parent CA on each hub.** Rejected
  because:
  - Adds a second trust chain alongside Vault for what is conceptually one
    secret-management surface.
  - Hub-dr DR rebuild path is harder: cert-manager's CA secret on hub-dc
    is also needed on hub-dr, requiring an out-of-band copy mechanism, which
    Vault already provides cleanly.
  - Doesn't fit ADR-0004's Vault-first principle.

- **Self-signed per cluster (no shared CA).** Rejected because:
  - On hub-dr DR rebuild, every agent must re-enroll against the new
    self-signed CA — that's a significant operational step in the middle of
    a DR drill.
  - No improvement over the simpler ACM-pull pattern we're moving away from
    (which at least doesn't require PKI bootstrap).
